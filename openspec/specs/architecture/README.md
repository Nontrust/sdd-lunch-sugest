# 아키텍처 가이드 (Architecture Guide)

## 한 줄 요약

> **4-레이어 단방향 의존 + 팩토리 함수 DI + 도메인 단위 서비스 분리**

> **이 문서는 현재 시점의 최선이지, 영구적 규범이 아니다.**
> 더 나은 방법이 있으면 바꾼다. 리팩토링 사이클마다 이 문서를 리뷰한다.

---

## 레이어 구조

```
┌─────────────────────────────────────────────┐
│  UI 레이어                                   │
│  app/**/page.tsx, 컴포넌트                    │
│  훅: useApi, useApiMutation                  │
│  경로 상수: lib/api/routes.ts                │
├─────────────────────────────────────────────┤
│  API 레이어                                  │
│  app/api/**/route.ts                         │
│  withHandler 래퍼로 에러 자동 처리             │
│  입력 검증 담당                               │
├─────────────────────────────────────────────┤
│  서비스 레이어                                │
│  lib/*/service.ts (팩토리 함수)               │
│  비즈니스 로직 + 오케스트레이션                │
├─────────────────────────────────────────────┤
│  데이터 레이어                                │
│  lib/storage/types.ts (인터페이스)            │
│  lib/storage/file-storage.ts (파일 구현)      │
└─────────────────────────────────────────────┘
```

### 핵심 규칙: 의존 방향은 위→아래만

- UI → API → 서비스 → 데이터
- **역방향 금지**: 서비스가 API를 import하거나, 데이터가 서비스를 import하면 안 된다
- **같은 레이어 간**: 팩토리 인자로 주입 (예: recommend 서비스가 history 서비스를 인자로 받음)

## 서비스 구현 패턴

### 파일 구조 (모든 서비스 동일)

```
lib/{domain}/
  types.ts      ← 서비스 인터페이스 정의
  service.ts    ← createXxxService(repo) 팩토리 함수
  index.ts      ← 기본 인스턴스 + 함수 re-export
```

### 왜 팩토리 함수인가?

| 대안 | 왜 안 쓰는가 |
|------|-------------|
| 클래스 | 프로젝트 규모에 비해 과도한 추상화. this 바인딩 문제 |
| 직접 import | 스토리지 교체/테스트 모킹 불가 |
| **팩토리 함수** | **함수형 스타일 유지 + DI 가능 + 최소 보일러플레이트** |

### 하위 호환성

`index.ts`에서 기본 인스턴스를 생성하고 함수를 re-export하므로, API 라우트는 변경 없이 사용 가능:

```typescript
// API 라우트에서 — 변경 없음
import * as restaurant from "@/lib/restaurant";
restaurant.getAll();
restaurant.add("새식당", "korean");
```

## 서비스 분리 기준

### 현재 도메인 서비스

| 서비스 | 도메인 엔티티 | 책임 범위 |
|--------|-------------|----------|
| restaurant | Restaurant | 식당 CRUD |
| menu | Menu | 메뉴 CRUD (식당의 하위 자원) |
| history | HistoryRecord | 추천 기록 저장/조회 |
| recommend | (조합) | 추천 로직 + 히스토리 오케스트레이션 |

### 새 서비스를 만들어야 할 때

1. 새 기능이 **기존 서비스의 도메인 범위를 벗어남**
2. 기존 서비스에 메서드 1-2개 추가로는 해결 안 됨
3. 독립적인 데이터 저장소가 필요함

### 오케스트레이션 원칙

여러 서비스를 조합하는 로직은 **주된 도메인 서비스**에 둔다:

```
추천 + 기록 저장 → recommend.recommend() (O)
API 라우트에서 pick + addRecord 조합 (X)
```

## 에러 처리 컨벤션

### 에러 코드 매핑

| 코드 | 의미 | 팩토리 | 사용 예 |
|------|------|--------|---------|
| 400 | 잘못된 입력 | `apiError(400, msg)` | 필수 필드 누락 |
| 404 | 리소스 없음 | `notFound("식당")` | 존재하지 않는 ID |
| 409 | 중복 | `duplicate("맛집")` | 같은 이름 식당 |
| 500 | 서버 오류 | (자동) | 예상치 못한 예외 |

### 에러 흐름

```
서비스에서 throw → withHandler가 catch → { error: "메시지" } JSON 응답
```

- **API 라우트**: try/catch 직접 작성 금지. `withHandler` 래퍼가 처리
- **서비스**: 비즈니스 규칙 위반 시 `apiError`/`duplicate`/`notFound`로 throw
- **프론트엔드**: `useApi`/`useApiMutation` 훅이 `{ error }` 필드를 자동 추출

### 에러 메시지 규칙

- 모든 에러 메시지는 **한국어**
- 사용자에게 보여줄 수 있는 수준의 문장 (개발 용어 X)
- 패턴: `"{대상}을(를) {동작}할 수 없습니다"` 또는 구체적 사유 명시

## 프론트엔드 패턴

### API 호출

| 유형 | 훅 | 반환값 |
|------|-----|--------|
| GET (데이터 조회) | `useApi<T>()` | `{ data, error, loading, fetch }` |
| POST/DELETE (변경) | `useApiMutation()` | `{ execute, error }` |

- API 경로는 반드시 `API` 상수 사용 (`lib/api/routes.ts`)
- 문자열 리터럴 `/api/restaurants` 직접 사용 금지

### 컴포넌트 분리 기준

| 신호 | 조치 |
|------|------|
| `page.tsx` 120줄 초과 | 하위 컴포넌트 분리 |
| 2개 이상 독립적 UI 영역 | 역할별 컴포넌트로 분리 |
| props drilling 2단계 초과 | Context 도입 검토 (아직 해당 없음) |

분리 후 `page.tsx`는 **상태 관리 + 컴포넌트 조합만** 담당.

### 타입 규칙

- 도메인 타입은 `lib/types/index.ts`에서만 정의
- 페이지 내 동일 타입 재정의 금지
- 서비스 인터페이스는 각 서비스의 `types.ts`에 정의

## 입력 검증 전략

```
외부 입력 → API 라우트에서 검증 → 서비스는 유효한 입력 전제
```

| 검증 종류 | 위치 | 예시 |
|-----------|------|------|
| 입력 형식 | API 라우트 | 필수 필드 존재, 타입 확인 |
| 비즈니스 규칙 | 서비스 | 중복 이름, 존재하지 않는 ID |

서비스에서 입력 형식을 다시 검증하지 않는다 (단일 책임 원칙).

## 스토리지 제약과 향후 전환

### 현재: 파일 기반 JSON

- `data/restaurants.json`, `data/history.json`
- 단일 사용자 전제 (동시성 안전하지 않음)
- ID 생성: `Date.now().toString()`

### 향후 DB 전환 시

리포지토리 인터페이스 덕분에 서비스 코드 변경 없이 전환 가능:

```typescript
// 현재
const service = createRestaurantService(createFileRestaurantRepository());

// DB 전환 시
const service = createRestaurantService(createPgRestaurantRepository(pool));
```

변경 범위: `lib/storage/` + 각 서비스의 `index.ts`만.

## 리팩토링 판단 기준 (코드 스멜 체크리스트)

리팩토링 사이클에서 아래 항목을 점검한다:

| 코드 스멜 | 조치 |
|-----------|------|
| 중복 코드 (3회 이상 반복) | 공유 모듈/훅으로 추출 |
| 비대한 파일 (120줄 초과) | 역할별 분리 |
| 타입 재정의 | `lib/types`로 통일 |
| 하드코딩된 문자열 (API 경로 등) | 상수로 추출 |
| 관심사 혼재 (API 라우트에 비즈니스 로직) | 서비스로 이동 |
| 역방향 의존 | 의존 방향 수정 또는 인터페이스 주입 |
| 에러 시 상태 미해제 | 훅으로 통합 또는 finally 블록 추가 |

### 분리 vs 통합 판단

- **분리**: 파일 120줄 초과, 독립적 책임 2개 이상, 재사용 가능성 있음
- **통합**: 유사한 코드 3줄 이하, 한 곳에서만 사용, 분리 시 오히려 이해도 저하
- **원칙**: "3줄 유사 코드가 성급한 추상화보다 낫다"

## 아키텍처 리뷰 (리팩토링 사이클마다 수행)

이 아키텍처는 **현재 프로젝트 규모와 요구사항**에 맞게 설계되었다. 프로젝트가 성장하면 당연히 바뀌어야 한다. 리팩토링 사이클마다 아래 체크리스트를 점검한다.

### 체크리스트

| # | 질문 | 부적합 신호 |
|---|------|-----------|
| 1 | 현재 레이어 구조가 새 피처를 자연스럽게 수용하는가? | 새 피처가 레이어를 우회하거나 억지로 끼워 맞춰야 함 |
| 2 | 서비스 간 의존이 단순한가? | 3개 이상 서비스를 주입해야 하는 팩토리가 생김 |
| 3 | 팩토리 함수 패턴이 생산성을 돕는가? | types.ts/service.ts/index.ts 보일러플레이트가 실제 로직보다 많음 |
| 4 | 파일 기반 스토리지가 충분한가? | 동시성 문제 실제 발생, 데이터 크기 성능 이슈 |
| 5 | 에러 처리 패턴이 충분한가? | 새로운 에러 유형이 기존 코드 매핑에 안 맞음 |
| 6 | 프론트엔드 훅이 요구사항을 커버하는가? | useApi/useApiMutation으로 안 되는 패턴 반복 (웹소켓, SSE 등) |
| 7 | 업계에 더 나은 접근법이 있는가? | 프레임워크 업데이트, 새 라이브러리, 커뮤니티 모범 사례 변화 |

### 리뷰 결과 기록

- **변경 필요**: proposal에 변경 사유 + 대안을 명시하고, 이 스펙과 README를 갱신한다
- **현행 유지**: proposal에 "아키텍처 리뷰 완료 — 현행 유지" 한 줄 기록

### 진화 사례 (예시)

아래는 향후 발생할 수 있는 아키텍처 전환 시나리오다:

| 현재 | 전환 시점 | 대안 |
|------|----------|------|
| 파일 기반 JSON | 다중 사용자, 동시성 필요 시 | SQLite → PostgreSQL (리포지토리 인터페이스로 무중단 전환) |
| 팩토리 함수 DI | 서비스 의존성 3개 이상 복잡해질 때 | DI 컨테이너 (tsyringe 등) 도입 검토 |
| useApi/useApiMutation | 실시간 업데이트 필요 시 | React Query / SWR 또는 WebSocket 훅 도입 |
| withHandler 래퍼 | 미들웨어 체인 필요 시 | Next.js middleware 또는 별도 미들웨어 파이프라인 |
| Date.now() ID | 다중 사용자 환경 | nanoid / UUID |
