# 리팩토링 사이클 (Refactoring Cycle Hook)

## 한 줄 요약

> **피처 2개 완료 → 리팩토링 1회 실행** — 기술 부채가 쌓이기 전에 정리한다.

## 왜 이 규칙이 있는가?

피처를 빠르게 구현하다 보면 중복 코드, 비대한 모듈, 관심사 혼재 등의 기술 부채가 자연스럽게 쌓인다. 부채를 방치하면 이후 피처 개발 속도가 점점 느려진다.

이 프로젝트에서는 **2개 피처가 완료될 때마다 리팩토링 사이클을 실행**하여 부채를 조기에 정리한다. "나중에 한꺼번에 정리하자"가 아니라, 작은 단위로 자주 정리하는 전략이다.

## 동작 방식

```
피처 1 완료 → 피처 2 완료 → 🔧 리팩토링 → 피처 3 완료 → 피처 4 완료 → 🔧 리팩토링 → ...
```

### 트리거 조건

- `openspec/changes/archive/`에 **피처 변경 2개**가 쌓이면 발동
- 리팩토링 변경 자체는 피처 카운트에 포함하지 않음

### 카운트 예시

| 순서 | 변경 | 유형 | 피처 카운트 |
|------|------|------|------------|
| 1 | category-menu-recommender | 피처 | 1 |
| 2 | history-smart-recommend | 피처 | 2 → **리팩토링 트리거!** |
| 3 | refactor-and-docs | 리팩토링 | (카운트 안 함) |
| 4 | (다음 피처) | 피처 | 3 |
| 5 | (다음 피처) | 피처 | 4 → **리팩토링 트리거!** |

## 리팩토링 사이클 진행 방법

1. **아키텍처 리뷰** (매 사이클 필수):
   - 현재 레이어 구조가 새로운 피처를 수용하기에 적합한가?
   - 서비스 간 의존 관계가 복잡해지고 있지 않은가?
   - 현재 패턴(팩토리 함수, 리포지토리 등)이 오히려 장애물이 되고 있지 않은가?
   - 업계에서 더 나은 접근법이 등장했는가?
   - 결론: 변경 필요 시 proposal에 사유/대안 명시, 불필요 시 "현행 유지" 기록
2. **코드 스멜 식별**: 직전 2개 피처에서 발생한 기술 부채 나열
   - 중복 코드, 비대한 파일, 타입 불일치, 관심사 혼재, 하드코딩 등
3. **proposal 작성**: Why에 트리거 피처 2개 + 아키텍처 리뷰 결론 명시
4. **openspec 프로세스 준수**: proposal → design → tasks → specs (일반 피처와 동일)
5. **하위 호환성 유지**: API 동작과 UI 기능 변경 없음. 내부 구현만 개선
6. **스펙 갱신**: 아키텍처 변경이 있으면 `openspec/specs/architecture/` 스펙도 함께 업데이트
7. **검증**: 빌드 성공 + 페이지별 동작 확인

## 리팩토링 이력

### 1회차 (피처 2개 후)

**트리거 피처**: `category-menu-recommender`, `history-smart-recommend`

| 리팩토링 | 영역 | 주요 내용 |
|----------|------|----------|
| refactor-and-docs | 프론트엔드 | useApi/useApiMutation 훅, 컴포넌트 분리, 타입 통일, API 경로 상수 |
| service-layer-interface | 백엔드 | RestaurantRepository/HistoryRepository 인터페이스, 서비스 팩토리 함수, recommend 오케스트레이션 이동 |

## 관련 파일

- `openspec/config.yaml` — 프로젝트 규칙에 "2개 피처 완료마다 리팩토링 사이클 실행" 명시
- `openspec/specs/refactoring-cycle/spec.md` — 정식 요구사항 명세
- `openspec/changes/archive/` — 완료된 피처/리팩토링 변경 이력
